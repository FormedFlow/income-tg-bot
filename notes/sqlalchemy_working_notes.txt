The argument to ForeignKey is most commonly a string of the form <tablename>.<columnname>, or for a table in a remote schema or “owner” of the form <schemaname>.<tablename>.<columnname>. It may also be an actual Column object, which as we’ll see later is accessed from an existing Table object via its c collection:
ForeignKey(user.c.user_id)
The advantage to using a string is that the in-python linkage between user and user_preference is resolved only when first needed, so that table objects can be easily spread across multiple modules and defined in any order.

Асинхронная алхимия
Как устроен цикл работы в асинхронном режиме при использовании core-like подхода:
1. Получаем async engine:
from sqlalchemy.ext.asyncio import create_async_engine

async_engine = create_async_engine(db_url)

2. Используем контекстные менеджеры 
async with async_engine.connect() либо async with async_engine.begin(), который сразу начинает транзакцию

3. Внутри используем
await async_conn.execute(...)

4. Контекстный менеджер закрывается, транзакция коммитится. Profit



ORM подход
1. Так же создаём движок
2. Получаем объект асинхронной сессии
from sqlalchemy.ext.asyncio import async_sessionmaker

engine = create_async_engine(db_url)
async_session = async_sessionmaker(engine, expire_on_commit=False) # Избавляемся от проблемы с внезапными синхронными запросами при получении доступа к expired атрибутам. Тут же отмечу что как-то нужно будет перенастроить loading strategy для relationships.

3. Открываем контекстный менеджер для работы с сессией, затем контекстный менеджер для транзакции:
async with async_session() as session:
    async with session.begin():
        await ...

4. Контекстный менеджер транзакции схлопывается, происходит коммит

1. Establishing Connectivity - the Engine
The object which is used for establishing connection with the database server is called the engine.
Engine is created by using function create_engine(db_url) from sqlalchemy package.
Engine object besides other properties acts like a connection pool.


2. Working with Transactions and the DBAPI
As often mentioned throughout each part of the tutorial, there are two ways of operating sqlalchemy in general: the core, and the ORM way.
While core functionality is mostly outside of my spectrum, let's focus on ORM tricks.
The main object which provides a way of executing, for example, plain sql statements is named Session. Usually, the process of working with Session object looks like this:

with Session(engine) as session:
    session.execute(statement=stmt)
    session.commit()

It should be noted that core-way counterpart of Session, Connection class, is very similar to that of Session.
In most cases, Session object refers to Connection under the hood.
The important thing is Session doesn't hold on to single Connection object, while executing several SQL statements, it gets a new one from connection pool each time.

Use 00_session.py for reference.


3. Working with Database Metadata
At first, let's talk about the Core approach for tables creation process.

Each declared table is stored in MetaData object in a dictionary-like fashion. MetaData is somewhat a parent container for tables, it has several methods allowing it to perform operations like CREATE or DROP on all the tables it contains. Those two specifically are:

metadata_obj = MetaData()
metadata_obj.create_all()
metadata_obj.drop_all()

MetaData object is used directly like this only in the Core approach.
Once MetaData object is created, we can define table Classes, inside which it is nessessary to reference MetaData object. 
The main things that should be pointed out is that the constrains are nothing but additional parameters for the Column class.

For reference use 01_table.py

Then, the ORM approach does the same things in a different way.

Actually, just take a look at 02_table_orm.py


Существует несколько способов из уже созданной в базе данных таблицы получить объект класса Table, с помощью приёма называемого table reflection.
Простейший способ в рамках core подхода сгенерировать объект метаданных и на его основе Table:

engine = create_engine(conn_string)
metadata_obj = MetaData()
user_table = Table('user', metadata_obj, autoload_with=engine)

user_table в итоге вернёт такой же объект как если бы мы создавали таблицу вручную с помощью классов Table, Column и т.д.


04. Working with data
Короче буду выносить сюда ключевые заметки.
Для создания INSERT statement используется функция insert из пакета sqlalchemy, есть два варианта использования, сразу с указанием параметров:
stmt = insert(user_table).values(name='фел габибов', age=28)

И второй вариант, без указания параметров в values, вместо этого передаём их в session/connection.execute:

with Session(engine) as session:
    session.execute(insert(user_table), 
    [
        {'name': 'бабинский феликс', age: 28},
        {'name': 'бабинс', age: 28}
    ])
    session.commit()


Можно использовать вложенный запрос, который соответствует в SQL такому выражению
INSERT INTO address (user_id, email_address)
SELECT id, name + '@gmail.com'
FROM address;

В sqlalchemy будет выглядеть так:
select_stmt = select(user_table.c.id, user_table.c.name + '@gmail.com')
insert_stmt = insert(address_table).from_select(
    ["user_id", "email_address"],
    select_stmt
)

Ну и далее session.execute(insert_stmt) и т.д.

Чтобы сделать SELECT id AS user_id, в SQLAlchemy используется метод .label('user_id'). Естественно, в результирующих строках потом можно обращаться к атрибуту .user_id

Думаю, стоит запомнить метод .returning соответствующий оператору RETURNING в SQL, который поддерживается не во всех диалектах. Позволяет при вставке вернуть заданные поля последней вставленной записи.
insert_stmt.returning(address_table.c.id, address_table.c.email_address)

Когда используем select с ORM-mapped классами, в виде select(User), в результате получаем строки, содержащие один элемент - экземпляр класса User. Чтобы сразу получить этот экземпляр, не в виде одноэлементного кортежа, а в виде экземпляра класса нужно использовать
session.scalars(select(User))

По аналогии с тем, как обычный текст может быть вставлен в SELECT в SQL, того же можно добиться в SQLAlchemy, используя text() из пакета SQLAlchemy:
select_stmt = select(text('Габибов Феликс'), user_table.c.name)

Существует так же аналогичная функция literal_column("'some text'") к которой потом можно применить .label() и обращаться как к отдельной колонке в result.


В .where() доступны операторы or_() и and_(). В случае где нужен and можно также просто перечислить условия через запятую. Не думаю, что это прямо важно, но SQLAchemy в SQL инструкциях, в частности в WHERE не ставит скобки, там где это не нужно, полагаясь на приоритет операторов.


Для простых проверок на равенство значений в колонках литеральным значениям существует "популярный" метод .filter_by(). Вроде как применяется к самой дальней слева таблице во FROM, либо к последней присоединенной JOINом. Использование выглядит так:
print(select(User).filter_by(name="феликс", fullname="феликс габибов"))
РАБОТАЕТ ТОЛЬКО ДЛЯ ORM сущностей. Именованные аргументы маппятся на имена атрибутов модели либо на сами имена колонок таблицы (согласно документации)


Насчёт джойнов, есть два метода - .join_from() и обычный .join()
.join_from() позволяет указать обе стороны (таблицы) которые джойним, .join() только правую, левая подхватывается сама из колонок селекта.
В обоих случаях условие ON так же генерируется само на основе внешних ключей, но может при необходимости и быть указано явно, с помощью дополнительного аргумента в .join() или .join_from() с использованием синтаксиса, аналогичному используемому в .where().
select(user_table.c.fullname).join(address_table, address_table.c.user_id == user_table.c.id)


Если по каким-то причинам нас не устраивает автоматически подхваченное FROM из колонок в .select(), можно вручную использовать 
.select_from().
А, ну и одна из таких возможных причин - недостаточная информация в .select(), например при использовании функции COUNT(*):

from sqlalchemy import func
print(select(func.count("*")).select_from(user_table))



Если обычный .join() или .join_from() по дефолту - это обычный INNER JOIN, то есть так же возможность сделать LEFT OUTER JOIN и FULL OUTER JOIN. Делается это с помощью ключевых аргументов isouter и full. Да, флаг isouter означает LEFT OUTER JOIN, вот такая замечательная семантика.
В SQLAlchemy нет RIGHT JOIN, при необходимости нужно просто менять таблицы местами.


.order_by() доступно как в Core, так и в ORM варианте:
select(user_table).order_by(user_table.c.name.asc())
select(User).order_by(User.name.desc())
Доступны .asc() .desc() тоже в обоих вариантах для ColumnElement, в случае если используем строковый вариант названия колонки или label - используем функции asc() и desc() из пакета sqlalchemy.


.group_by() и .having() доступны, по всей видимости как для Core подхода, так и для ORM-based. Всё стандартно.
В пакете sqlalchemy есть модуль func предоставляющий доступ к различным SQL функциям, в частности к аггрегирующей функции COUNT()
Пример с .group_by()

with engine.connect() as conn:
    result = conn.execute(
        select(User.name, func.count(Address.id).label("count"))
        .join(Address)
        .group_by(User.name)
        .having(func.count(Address.id) > 1)
    )
    print(result.all())


В .group_by() и .order_by() можно передавать имена колонок или алиасы с целью не вычислять уже готовое выражение заново.
В таком случае в .group_by() или/и .order_by() передаётся в текстовом виде название колонки или алиас, но он не подставляется "как есть" в итоговое выражение, поскольку это бы создавало риск SQL-инъекции, а сперва вычисляется значение в .select() и затем с ним сопоставляется текст из .group_by()/.order_by(). В случае ненахождения в селекте такого названия поднимается ошибка. 
В .order_by() можно передавать аргумент
Пример использования:

from sqlalchemy import func, desc
stmt = (
    select(Address.user_id, func.count(Address.id).label("num_addresses"))
    .group_by("user_id")
    .order_by("user_id", desc("num_addresses"))
)
print(stmt)


Внезапно выяснилось что алиасы в рамках документации SQLAlchemy это немного другое, а именно метод Table.alias(), используемый, в частности для того, чтобы при использовании в разделе запроса FROM одинаковых таблиц несколько раз, мы бы могли сослаться на их алиасы:

user_alias_1 = user_table.alias()
user_alias_2 = user_table.alias()
print(
    select(user_alias_1.c.name, user_alias_2.c.name).join_from(
        user_alias_1, user_alias_2, user_alias_1.c.id > user_alias_2.c.id
    )
)

Для ORM сущностей есть свой вариант создания алиаса - функция aliased(). Под капотом создаёт объект Alias() для связанного объекта Table()
Пример (не очень понял к слову):

from sqlalchemy.orm import aliased
address_alias_1 = aliased(Address)
address_alias_2 = aliased(Address)
print(
    select(User)
    .join_from(User, address_alias_1)
    .where(address_alias_1.email_address == "patrick@aol.com")
    .join_from(User, address_alias_2)
    .where(address_alias_2.email_address == "patrick@gmail.com")
)


.subquery() и .cte() - методы с помощью которых можно создать подзапрос или Common Table Expression.
subq = (
    select(func.count(address_table.c.id).label("count"), address_table.c.user_id)
    .group_by(address_table.c.user_id)
    .subquery()
)

И потом на него можно ссылаться как на subq, к его колонкам subq.c.count и т.д. При использовании обоих методов, подзапросы на уровне SQL получают "анонимные" алиасы:

WITH anon_1 AS
(SELECT count(address.id) AS count, address.user_id AS user_id
FROM address GROUP BY address.user_id)
 SELECT user_account.name, user_account.fullname, anon_1.count
FROM user_account JOIN anon_1 ON user_account.id = anon_1.user_id

Можно изменять эти алиасы, передавая необходимое название в качестве первого аргумента в методы .subquery() и .cte()



При работе с подзапросами в рамках ORM подхода требуется использовать aliased(Class, subquery).
Подзапросы, составленные с помощью .subquery() и .cte() можно промаппить на ORM класс, на базе которого эти запросы построены, для того чтобы можно в последствии было бы их использовать уже в обобщённом запросе вместе с другими ORM классами:

subq = select(Address).where(~Address.email_address.like("%@aol.com")).subquery()
address_subq = aliased(Address, subq)
stmt = (
    select(User, address_subq)
    .join_from(User, address_subq)
    .order_by(User.id, address_subq.id)
)
with Session(engine) as session:
    for user, address in session.execute(stmt):
        print(f"{user} {address}")




Дальше .scalar_subquery(). Скалярный подзапрос и correlated subquery. Скалярный запрос - запрос, возвращающий 0 или 1 строку с ровно одним полем, для использования в .select() как колонки, либо (наверное) в where().
C коррелирующим запросом происходит некая магия, если он ссылается сам по себе на две таблицы, а потом его запихивают внутрь запроса, где на одну уже ссылаются, то в SQL инструкции эта общая таблица опускается во вложенном запросе, поскольку она есть снаружи.
Crazy, но это работает в SQL.
Пример:
subq = (
    select(func.count(address_table.c.id))
    .where(user_table.c.id == address_table.c.user_id)
    .scalar_subquery()
)
print(subq)

(SELECT count(address.id) AS count_1
FROM address, user_account
WHERE user_account.id = address.user_id)

При использовании в другом запросе:
stmt = select(user_table.c.name, subq.label("address_count"))
print(stmt)

SELECT user_account.name, (SELECT count(address.id) AS count_1
FROM address
WHERE user_account.id = address.user_id) AS address_count
FROM user_account

Упоминание таблицы user_account во вложенном запросе исчезло!



В сложных случаях с джойнами(?), когда непонятно с какой таблицей запрос коррелировать, нужно указать это самому, иначе получим ошибку.
Короче указывать таблицу, которая будет вовне нужно с помощью .correlate(Model).

Дальше о функциях. Многие SQL функции доступны из фабрики функций sqlalchemy.func.
SQLAlchemy автоматически рэндерит распространенные функции вроде NOW() в соответствующем текущему диалекту виде.
Вот так можно посмотреть скомплированные инструкции для определенного диалекта:
from sqlalchemy.dialects import postgresql
print(select(func.now()).compile(dialect=postgresql.dialect()))
from sqlalchemy.dialects import oracle
print(select(func.now()).compile(dialect=oracle.dialect()))

Любое название, указанное в пространстве имён func.namespace автоматически преобразуется в SQL-функцию с соответствующим названием.
print(select(func.some_crazy_function(user_table.c.name, 17)))

SELECT some_crazy_function(user_account.name, :some_crazy_function_2) AS some_crazy_function_1
FROM user_account

Дальше в документации говорится о том что SQL функции возвращают некие данные, если в python коде мы говорим о return types, то возвращаемые SQL выражениями типы называют SQL return types

# pre-configured SQL function (only a few dozen of these)
func.now().type # DateTime()

# arbitrary SQL function (all other SQL functions)
func.run_some_calculation().type # NullType()

Возвращаемый тип для функций можно указывать явно с помощью именованного аргумента type_=, которому передаётся дочерний класс TypeEngine либо экземлпяр класса.
from sqlalchemy import JSON
function_expr = func.json_object('{a, 1, b, "def", c, 3.5}', type_=JSON)




Для приведения типов, или просто явного указания необходимого типа данных в неоднозначных ситуациях, используется функция cast(), транслирующуяся в соответствующий оператор CAST в SQL.
from sqlalchemy import cast
stmt = select(cast(user_table.c.id, String))
with engine.connect() as conn:
    result = conn.execute(stmt)
    result.all()

BEGIN (implicit)
SELECT CAST(user_account.id AS VARCHAR) AS id
FROM user_account
[...] ()

В объекте Result мы тоже получаем соотвтествующий python тип.



И есть аналог cast() - type_coerce(), который помечает тип данных лишь на python-стороне, т.е. не рэндерит оператор CAST в SQL.
Сперва не особо понял зачем это, но пример работы с JSON внёс ясность. 
import json
from sqlalchemy import JSON
from sqlalchemy import type_coerce
from sqlalchemy.dialects import mysql
s = select(type_coerce({"some_key": {"foo": "bar"}}, JSON)["some_key"])
print(s.compile(dialect=mysql.dialect()))

SELECT JSON_EXTRACT(%s, %s) AS anon_1

Здесь, благодаря тому, что мы явно указали тип данных выражения с помощью type_coerce() была использована необходимая SQL функция JSON_EXTRACT().

UPDATE и DELETE
Если функция insert() реализующая вставку строк в таблицу может быть использована только в рамках Core-подхода, т.е. с использованием Table, то update() и delete(), несмотря на то, что для ORM варианта есть специальный более удобный синтаксис, могут быть использованы как для Core, так и для ORM подхода.

Базовый update(), с одной таблицей и одной изменяемой строкой:
from sqlalchemy import update
stmt = (
    update(user_table)
    .where(user_table.c.name == "patrick")
    .values(fullname="Patrick the Star")
)
print(stmt)

UPDATE user_account SET fullname=:fullname WHERE user_account.name = :name_1

Изменяемые поля указываются в .values() в виде именованных параметров, обозначающих имена колонок. Чтобы обновить сразу несколько строк, нужно использовать функцию sqlalchemy.bindparam(), применяем её там, куда будем принимать параметры:
from sqlalchemy import bindparam
stmt = (
    update(user_table)
    .where(user_table.c.name == bindparam("oldname"))
    .values(name=bindparam("newname"))
)
with engine.begin() as conn:
    conn.execute(
        stmt,
        [
            {"oldname": "jack", "newname": "ed"},
            {"oldname": "wendy", "newname": "mary"},
            {"oldname": "jim", "newname": "jake"},
        ],
    )

BEGIN (implicit)
UPDATE user_account SET name=? WHERE user_account.name = ?
[...] [('ed', 'jack'), ('mary', 'wendy'), ('jake', 'jim')]
<sqlalchemy.engine.cursor.CursorResult object at 0x...>
COMMIT


Можно использовать в связке с correlated subquery():
scalar_subq = (
    select(address_table.c.email_address)
    .where(address_table.c.user_id == user_table.c.id)
    .order_by(address_table.c.id)
    .limit(1)
    .scalar_subquery()
)
update_stmt = update(user_table).values(fullname=scalar_subq)
print(update_stmt)


UPDATE user_account SET fullname=(SELECT address.email_address
FROM address
WHERE address.user_id = user_account.id ORDER BY address.id
LIMIT :param_1)



UPDATE нескольких таблиц одновременно - фича mySQL, в других популярных СУБД такая операция не предусмотрена.
Однако, в PostgreSQL, MSSQL и mySQL есть синтаксис для апдейта одной таблицы, используя другую для фильтрации:
update_stmt = (
    update(user_table)
    .where(user_table.c.id == address_table.c.user_id)
    .where(address_table.c.email_address == "patrick@aol.com")
    .values(fullname="Pat")
)
print(update_stmt)
